# 경력기술서



## 개요

웹 빌더 프로그램 **RENOBIT**의 프론트엔드 개발 및 제품 개발 방향을 담당하는 개발팀장입니다.
2020년 5월 신입으로 입사하여, 대규모 3D 데이터 시각화 프로젝트 수행, 제품 이슈 대응 체계 운영, 제품 아키텍처 리뉴얼 주도를 거쳐 2024년 11월 개발팀장이 되었습니다.

주요 기술 스택은 **JavaScript, THREE.js(WebGL 기반 3D 시각화)** 이며, 단순 요구사항 구현을 넘어 *“무엇을 만드는지 정확히 이해한 뒤 만든다”* 는 원칙과, *“불필요한 일을 제거하는 것도 일이다”* 라는 관점을 일관되게 적용해 왔습니다.

입사 후 수습 기간 동안 RENOBIT 제품과 업무 전반에 적응하였습니다.

## 2. 보일러/터빈 AI 데이터 웹 시각화 프로젝트 (2020.10 ~ 약 1년)

브라우저 기반 3D 시각화 애플리케이션을 개발했으며, 프론트엔드 개발을 담당했습니다.
- **기술 스택:** Vue 2, THREE.js, ECharts
- **도메인:** 보일러/터빈 AI 데이터의 누설 데이터 시각화

### 기술적 성취

- **대규모 3D 데이터 렌더링:** 수십만 건에 달하는 누설 데이터를 sphere geometry로 브라우저에 렌더링해야 했습니다. `draw call`과 `merge` 개념을 이해·적용하여 렌더링 성능 문제를 해결했습니다. THREE.js 공식 예제 중 유사 사례를 학습하고 프로젝트에 적용하는 방식으로 접근했습니다.
- **차트 인터랙션 구현:** 3D와 ECharts를 처음 다루는 상황이었으나, 공식 문서와 예제를 학습하며 요구사항을 구현했습니다. 사용자 인터랙션에 대응하는 `function formatter` 등 라이브러리가 제공하는 기능 범위를 적극적으로 탐색하여 요구사항에 맞췄습니다.

### 협업 및 일하는 방식

- 디자이너, PM, 백엔드 개발자, 고객사와 폭넓게 소통했습니다.
- 단순 구현에 그치지 않고 **개발 대상 자체를 깊이 이해**하려 했습니다. 이해한 내용을 글과 멘탈 모델로 정리해 협업 구성원 전체에게 공유함으로써, 서로 간 이해의 격차를 줄이고 결과물의 정확성을 높였습니다.
- **무조건 구현하지 않고, 필요성부터 검증**했습니다. 기능 명세가 주어지면 그 기능이 왜 필요한지를 먼저 파악하고, 정말 필요한지 논의했습니다.
  - *예시:* 복잡한 커스텀 차트 인터랙션 명세가 있었으나, 구현에 앞서 해당 인터랙션의 필요성을 논의하여 더 단순한 차트 + 팝업 구성으로 대체했습니다.

### 성과

약 1년간 진행한 프로젝트를 완수했으며, 사고 없이 계약이 정상 종료(대금 전액 수령)되었습니다.

## 3. 제품/프로젝트 이슈 게시판 운영 (약 2년 이상)

제품 개발과 병행하여, 회사가 수행하는 프로젝트·제품·구현 과정에서 발생하는 이슈를 처리하는 게시판의 메인 담당자로 2년 이상 운영했습니다. 제품에 익숙하지 않던 시기에도 제품에 대한 엄밀한 검토와 이슈 보고자와의 적극적 소통을 병행했습니다.

이 과정에서 정립한 이슈 대응 원칙은 다음과 같습니다.

- **이슈 분류의 명확화:** 제품 자체의 문제인지, 제품을 활용한 개발 과정의 문제인지를 분명히 구분했습니다.
- **근거 기반 커뮤니케이션:** 정중하고 간결한 메시지로 소통하되, 주장을 남길 때는 MDN, HTML Spec 등 공식 근거 링크를 첨부하여 신뢰성을 확보했습니다.
- **튜토리얼형 가이드 작성:** 비개발자도 따라 할 수 있는 튜토리얼 형식으로 작성하고, 상대가 이미 알고 있다고 전제하지 않고 모를 수 있다는 전제로 문서를 구성했습니다.
- **불필요한 작업 최소화:** 코드 수정·작업 없이 해결 가능한 이슈는, 작업을 최소화하는 방향을 함께 논의하여 도출했습니다.

이 경험을 통해 커뮤니케이션 방법, 이슈 발생 시 처리 절차와 우선순위, 공식 답변 및 가이드 작성 방법, 그리고 개발 환경·프로젝트 수행상의 어려움에 대한 이해를 폭넓게 쌓았습니다.

## 4. RENOBIT 제품 개발 방향 주도 및 리뉴얼 (2023.11 ~)

2023년 11월부터 RENOBIT의 실질적 개발 방향을 담당했습니다(공식 팀장 직책은 2024년 11월부터).
- **팀 구성:** 프론트엔드 본인 + 프론트엔드 주니어 1명, 백엔드 시니어 1명·주니어 1명. 경력이 많은 백엔드 시니어로부터 많은 도움을 받을 수 있었습니다.
- 기존 제품의 리뉴얼 작업을 시작했습니다.

이 시점부터 지시받는 업무·방향에 대해 본질을 의심하고 의견을 적극적으로 개진했습니다. 보고용으로 화려하거나 그럴듯해 보이는 작업보다, 현재 사업에 실제로 필요한 일에 집중하고자 했으며, **제품의 프로세스와 실체**에 초점을 두기 시작했습니다.

### 4-1. 문제 진단 — RENOBIT 구조와 SI 환경의 부정합

**RENOBIT의 구조**
- 2018년경부터 사용된, 폐쇄망 SI 사업 개발자를 위한 도구입니다. 빌드된 war를 프로젝트 현장으로 가져가 필요한 사이트를 개발하는 방식입니다.
- **페이지/컴포넌트** 개념을 가지며, 리소스(이미지·3D 모델링)와 데이터셋(자체 수집 프로그램 존재)을 등록해 컴포넌트에 연결하고, 에디팅을 통해 대시보드를 제작하는 것이 주요 패턴입니다.
- 사용자 인터랙션 제어는 스크립트 에디터를 통해 이루어졌습니다(HTML/CSS 없이 JS만 사용).
- 컴포넌트는 사실상 하나의 큰 클래스로, 그 메소드를 UI와 연결해 사용하도록 설계되었습니다. (예: 차트 컴포넌트를 배치 → 더블클릭 시 팝업 → 사전 등록 데이터셋 연결)

**드러난 문제**
- **컴포넌트 의도의 붕괴:** 장기간 개발과 담당자 교체, 요구사항 누적 속에서 클래스의 의도가 흐려지고, 코드 유지보수성·사용성이 점차 악화되었습니다.
- **SI 환경과의 근본적 부정합:** SI는 *어떤 요구사항이 올지 예측 불가능*하다는 점이 본질입니다. 정형화된 부분은 컴포넌트로 만들 수 있지만, 현장 요구사항은 다양하고 거의 모두 구현해 내야 합니다. 기존 컴포넌트로 구현이 불가능하거나 훨씬 복잡한 인터랙션이 필요한 순간, 컴포넌트는 커스텀이 불가피해지고 정리되지 않은 임시 코드로 전락했습니다. **결국 ‘편의를 위해 만든 컴포넌트’가 아무것도 없는 것보다 못한, 구현 자체가 제약이 되는 상황**이 발생했습니다.
- **데이터셋 중복 호출:** 컴포넌트마다 데이터셋을 연결하는 구조여서, 동일한 데이터셋이라도 차트가 10개면 10번 호출되었습니다.
- **컴포넌트의 실체:** 미리 만들어 둔 컴포넌트는 사실상 컴포넌트라기보다, 스크립트 에디터로 약속된 몇 개의 value만 바꿀 수 있는 ‘이미지’에 가까웠습니다. 그 value조차 애플리케이션 전용의 특수한 이름·데이터 구조를 가져 미경험 개발자가 이해하기 어려웠고, 매뉴얼도 사용법을 충분히 설명하지 못했습니다.
- **디자인팀의 직접 마크업:** 가장 큰 문제는 컴포넌트 배치를 디자인팀이 수행했다는 점입니다. 디자인팀은 컴포넌트를 본래 목적대로가 아니라 일종의 ‘그림’으로 화면에 배치했고, 개발자는 그 그림에 스크립트를 붙여 프로젝트를 수행했습니다. 결과적으로 고정된 단일 대시보드만을 위한 작업이 반복되었고, HTML·CSS·JS 작업이 올바르게 분리되지 못했습니다.

**누적된 결과**
- 프로젝트 수행 건수는 늘어도 **재사용 가능한 소스코드는 축적되지 않는 구조**였습니다.
- 시대에 따라 개선되는 CSS 표준을 활용하기 어려웠고, 그림 붙이듯 구성한 화면 구조는 반응형에 대응할 수 없었습니다. 과거에는 문제되지 않던 것들이 점차 문제로 드러났습니다.

### 4-2. 핵심 의사결정

이 상황에서 다음과 같은 과감한 결정을 내렸습니다.

1. **기존 컴포넌트 전면 제거** — 대신 개발자가 마크업·스타일링·스크립트를 표준에 맞게 제약 없이 적용할 수 있는 환경을 구축한다. (HTML / CSS / JavaScript 작업 영역의 분리)
2. **디자이너의 직접 마크업 중단** — 디자이너는 RENOBIT 화면에 직접 마크업하지 않고, Figma 등 디자인 전용 도구를 적극 활용하는 방향으로 프로젝트를 수행한다.
3. **UI를 위해 기존 UI를 제거** — 제대로 된 UI를 만들기 위해, 역설적으로 기존의 UI를 모두 제거하고, 대시보드 아키텍처를 위한 유틸과 프레임워크를 개발한다.

### 4-3. 실행

- **에디터 교체:** RENOBIT의 에디터를 Ace Editor에서 VS Code의 Monaco Editor로 변경하는 작업에 착수했습니다.
- **라이브러리 최신화:** 변화하는 웹 개발 환경을 따라잡기 위해 라이브러리를 최신화했습니다. (예: THREE.js는 제품 사용 이후 한 번도 업데이트되지 않은 2018년 버전 r95 상태였습니다.)
- **컴포넌트 카테고리 표준화:** Bootstrap을 참고한 카테고리로 변경하여 표준적인 컴포넌트 목록을 구성했습니다.
- **Figma 도입 검증:** Figma Dev Mode를 구독하고, 실제 디자인팀의 Figma 화면을 받아 RENOBIT으로 퍼블리싱하는 과정에 직접 익숙해졌습니다. *“내가 먼저 사용해 가능성을 판단해야 이 방향을 주장할 명분이 생긴다”* 는 판단에 따른 것입니다.
- **아키텍처 정립:** 개발자마다 다른 프로젝트 구축 패턴을 구조화하기 위해 RENOBIT 전용 아키텍처 설계에 착수했습니다. (상세는 아래 「핵심 아키텍처 설계」 절 참조)

## 핵심 아키텍처 설계 — RENOBIT 대시보드 프레임워크

RENOBIT 리뉴얼 과정에서 내린 결정들을 실제 구조로 구현하기 위해, 대시보드 제작을 위한 런타임 아키텍처를 직접 설계했습니다. 설계의 출발점은 역할의 단방향 분리였습니다. 페이지를 데이터·갱신 주기·이벤트를 통제하는 오케스트레이터로, 컴포넌트를 자신의 콘텐츠를 렌더링하는 수동적 구독자로 구분하고, 둘 사이의 접점을 토픽(topic) 문자열과 도메인 이벤트라는 최소한의 계약으로 한정했습니다. 요구사항을 예측할 수 없는 SI 환경에서는 컴포넌트가 데이터와 제어를 직접 움켜쥘수록 커스터마이징이 어려워진다고 보았기 때문에, 통제권을 페이지로 모으고 컴포넌트는 서로의 내부를 모른 채 느슨하게 결합되도록 한 것입니다.

데이터 흐름은 토픽 기반의 발행-구독(pub-sub) 구조로 설계했습니다. 페이지가 토픽과 데이터셋의 매핑을 등록하고 데이터를 발행하면, 그 토픽을 구독한 여러 컴포넌트에 한 번의 fetch 결과가 분배됩니다. 이를 통해 *‘차트 10개가 동일한 데이터셋을 각각 호출하던’* 기존의 중복 호출 문제를 구조적으로 제거했고, 컴포넌트는 데이터가 어디서 어떻게 오는지 알 필요 없이 토픽만 구독하면 되도록 했습니다.

```javascript
// GlobalDataPublisher — 한 번의 fetch 결과를, 그 토픽을 구독한 모든 컴포넌트에 분배
async fetchAndPublish(topic, page) {
    const { datasetName, param } = mappingTable.get(topic);
    const subs = subscriberTable.get(topic) || new Set();           // 이 토픽 구독자들
    const data = await Wkit.fetchData(page, datasetName, param);    // fetch는 한 번
    fx.each(({ instance, handler }) => handler.call(instance, data), subs);  // N개에 분배
}
```

이벤트 흐름은 경량 이벤트 버스와 공통 도구 계층으로 표준화했습니다. 2D는 이벤트 위임으로, 3D는 레이캐스팅으로 사용자 상호작용을 받아들이되, 그 결과를 DOM 이벤트가 아닌 도메인 이벤트로 승격시켰습니다. 컴포넌트는 *‘무슨 일이 일어났는가’* 만 발행하고, *‘그래서 무엇을 할 것인가’* 는 페이지가 결정하도록 분리하여, 컴포넌트의 재사용성과 페이지의 제어권을 양립시켰습니다.

리소스 관리에서는 생성과 정리의 1:1 대칭을 구조적으로 강제했습니다. 이벤트 핸들러, 구독, 데이터 갱신 타이머, 3D 리소스는 라이프사이클 종료 시점에 빠짐없이 역순으로 정리되도록 설계하여 누수를 차단했습니다. 특히 데이터 갱신 주기는, 통제권을 쥔 페이지(`loaded.js`)에서 `setInterval` 대신 `setTimeout` 체이닝으로 구현했는데, 이는 응답이 완료되기 전에 다음 요청이 누적되는 것을 방지하고, 갱신 중단 후 진행 중이던 요청이 타이머를 다시 등록하는 문제(좀비 타이머)를 가드로 막기 위한 선택이었습니다.

```javascript
// 페이지 loaded.js — 토픽별 폴링 (setInterval 미사용, setTimeout 체이닝)
this.startAllIntervals = () => {
    go(this.pageDataMappings, each(({ topic, refreshInterval }) => {
        if (!refreshInterval) return;
        const state = { _stopped: false, _timerId: null };   // 토픽별 상태
        this.pageIntervals[topic] = state;

        const scheduleNext = () => {
            if (state._stopped) return;                      // 중단됐으면 재등록 안 함 (좀비 가드)
            state._timerId = setTimeout(() => {
                fetchAndPublish(topic, this, this.pageParams[topic] || {})
                    .catch(err => console.error(`[Page] ${topic} refresh failed:`, err))
                    .finally(scheduleNext);                  // 응답이 끝난 뒤에야 다음 예약
            }, refreshInterval);
        };
        scheduleNext();
    }));
};

this.stopAllIntervals = () => {
    go(Object.entries(this.pageIntervals), each(([_, state]) => {
        state._stopped = true;                               // 진행 중 요청이 재등록 못 하게 차단
        clearTimeout(state._timerId);
    }));
};
```

전반의 기반 도구로는 함수형 자바스크립트 라이브러리(fx.js, 외부 라이브러리)를 채택하여 파이프라인과 비동기 처리를 일관되게 다루었고, 그 위에 명시적인 에러 처리 정책을 직접 설계했습니다. 유틸 계층은 에러를 복구하지 않고 그대로 전파(fail-fast)하되, 호출자인 페이지는 도메인 판단에 따라 격리(fail-safe)를 선택할 수 있도록 계층을 나누었습니다. 예를 들어 대시보드의 여러 위젯이 서로 독립적일 때는 한 토픽의 실패가 다른 위젯의 동작을 멈추지 않도록 개별적으로 격리하고, 인증처럼 의존 사슬이 있는 경우에는 fail-fast로 분리하는 기준을 정립했습니다.

이러한 런타임 아키텍처를 통해, 그림 붙이듯 구성하던 화면을 표준 기반의 마크업과 반응형 대응이 가능한 구조로 바꾸고, 데이터·이벤트·리소스의 통제권을 페이지로 일원화할 수 있었습니다.

## AI 에이전트를 통한 컴포넌트 재생산

대시보드 런타임이 자리를 잡은 뒤에는, 그 위에서 반복되는 생산 작업 자체를 다시 들여다봤습니다. 디자이너는 Figma에서 같은 컴포넌트의 디자인 변형을 빠르게 만들어 내지만, 그 각각을 동작하는 컴포넌트로 바꾸는 일은 결국 개발자의 손을 거쳐야 했고, 변형이 늘수록 개발자가 병목이 됐습니다. 이 병목을 구조적으로 풀기 위해, 컴포넌트 생산 자체를 AI 에이전트가 대신 수행할 수 있는 체계를 설계했습니다.

출발점은 디자인과 행동의 분리였습니다. Figma에서 나온 마크업과 스타일은 컴포넌트가 소유하고, 행동 로직은 Mixin이라는 재사용 가능한 기능 단위로 분리하여 약속된 CSS 선택자 계약을 통해서만 DOM에 접근하도록 했습니다. 각 Mixin은 고유한 네임스페이스로 인스턴스에 주입되고 서로의 존재를 모른 채 조립 코드에 의해서만 조합됩니다. 아래 Mixin은 *어떤 HTML 구조인지 전혀 모른 채*, 주입받은 선택자 계약(`cssSelectors`)만으로 데이터를 채웁니다 — 그래서 디자인이 바뀌어도 이 코드는 변하지 않습니다. (실제 구현을 핵심만 간추림)

```javascript
// FieldRenderMixin — 기능을 네임스페이스로 주입, DOM은 '선택자 계약'으로만 접근
function applyFieldRenderMixin(instance, { cssSelectors = {} }) {
    const ns = (instance.fieldRender = {});
    ns.cssSelectors = { ...cssSelectors };                  // 주입받은 선택자 보존
    ns.renderData = function ({ response: data }) {
        Object.entries(data).forEach(([key, value]) => {
            if (value == null || !cssSelectors[key]) return;
            const el = instance.appendElement.querySelector(cssSelectors[key]);
            if (el) el.textContent = value;                 // HTML 구조를 모른 채 계약으로만 접근
        });
    };
    ns.destroy = function () { ns.renderData = null; instance.fieldRender = null; };
}
```

이렇게 분리하고 나면, 디자인 변형 하나를 컴포넌트로 만드는 일은 *‘전달받은 마크업에 정해진 Mixin을 조립하고 선택자 계약을 잇는’* 기계적 절차로 환원됩니다. 컴포넌트의 행동 코드(`register.js`)는 사실상 *어떤 기능을 가져와 어떤 토픽에 연결하는가* 를 선언하는 조립표가 됩니다.

```javascript
// 컴포넌트 register.js — Figma 마크업은 그대로, '조립'으로 컴포넌트를 만든다
applyFieldRenderMixin(this, { cssSelectors: { headline: '.card__headline', supporting: '.card__supporting' } });
applyListRenderMixin(this,  { cssSelectors: { container: '.card__actions', template: '#card-action-template',
                                              actionLabel: '.card__action-label' } });

this.subscriptions = {                          // 토픽 → Mixin 메서드(함수 참조)
    cardInfo:    [this.fieldRender.renderData],
    cardActions: [this.listRender.renderData]
};
```

이 절차가 정형화되어 있다는 점이 곧 *사람이 매번 반복할 필요가 없는, 그래서 에이전트에게 맡길 수 있는 형태* 라는 의미입니다.

다만 핵심은 *‘AI에 일을 맡긴다’* 가 아니라, **에이전트가 흔들림 없이 같은 품질을 반복해 낼 수 있는 환경을 설계하는 일**이었습니다. 에이전트는 입력에 모호함이 남은 만큼 결과가 흔들리므로, 생산 라인을 설계하는 엔지니어가 공정의 변수를 하나씩 고정하듯 컴포넌트가 만들어질 자리 자체를 규격화하는 데 대부분의 공수를 들였습니다. 구체적으로는 세 가지를 갖췄습니다. 첫째, **구조의 표준화** — 모든 컴포넌트가 동일한 디렉터리·파일 구조(디자인 변형, 조립 코드, 정리 코드, 프리뷰)를 따르게 해 산출물의 형태를 예측 가능하게 했습니다. 둘째, **Mixin의 카탈로그화** — 각 Mixin이 무엇을 하는지, 어떤 선택자 계약을 요구하는지, 어떤 네임스페이스 메서드를 제공하는지를 문서로 명세해 에이전트가 추측 없이 필요한 기능을 고르고 조립하도록 했습니다. 셋째, **검증의 자동화** — 산출물이 규약을 지켰는지를 사람이 일일이 검토하는 대신, Hook이 단일 파일 검사와 컴포넌트·페이지 간 교차 검증을 자동으로 수행하게 하여 게이트를 통과한 결과만 유효한 것으로 인정했습니다.

그 결과 컴포넌트 생산의 정확성은 결과물을 사후에 점검해 확보하는 것이 아니라, *생산되는 자리 자체를 어긋날 수 없게 설계함으로써* 구조적으로 확보됐습니다. *‘프로젝트 수행 건수는 늘어도 코드는 쌓이지 않던’* 구조가, 디자인 변형이 아무리 쏟아져도 개발자가 병목이 되지 않고 컴포넌트가 일회성 그림이 아니라 일관된 품질의 재사용 가능한 자산으로 축적되는 라인으로 바뀐 것입니다.

## AI 에이전트 기반 생산 파이프라인 설계·운용

컴포넌트 재생산을 실제로 구동하기 위해, AI 코딩 에이전트(Claude Code) 환경 위에 생산 파이프라인 전체를 직접 설계하고 운용했습니다. 단순히 에이전트에게 작업을 시키는 수준이 아니라, *작업을 어떻게 분할하고, 컨텍스트를 어떻게 격리하며, 산출물의 품질을 무엇으로 보장할 것인가* 를 시스템으로 설계하는 것이 목표였습니다.

먼저 작업을 단계로 나눠 스킬(skill)로 모듈화했습니다. Figma 변환 → 컴포넌트화 → 페이지 조립 → 검토 → Mixin 개발이라는 생산 흐름의 각 단계를 독립된 스킬로 정의하고, 진입점 스킬이 상황을 판단해 하위 스킬을 호출하도록 구성했습니다. 같은 작업도 자율 수준에 따라 *승인 기반 수동 진행*과 *완전 자동 진행*으로 나눠, 신뢰가 검증되기 전에는 사람이 개입하고 검증된 뒤에는 손을 떼도록 했습니다.

대량 생산에서는 컨텍스트 한계가 병목이 되므로, 이를 서브에이전트 오케스트레이션으로 풀었습니다. 메인 루프가 다음 대상을 파악해 매 사이클 독립된 컨텍스트의 서브에이전트에게 *단일 컴포넌트 생산*만 위임하고, 완료된 요약만 회수해 커밋한 뒤 다음 대상으로 넘어가도록 했습니다. 각 사이클이 깨끗한 컨텍스트에서 시작되므로, 수십 개의 컴포넌트를 연속으로 생산해도 앞선 작업의 맥락이 누적되어 품질이 흔들리는 일을 막을 수 있었습니다. 각 스킬은 메타데이터와 메인 루프 절차로 정의되고, Phase 1에서 서브에이전트를 호출합니다.

```text
# 스킬 정의 (produce-standard-auto/SKILL.md)
---
name: produce-standard-auto
description: 2D Standard 컴포넌트를 서브에이전트 기반으로 한 컴포넌트씩
             독립 컨텍스트에서 완전 자동 생산한다
model: opus
---

[메인 루프]
  ├ Phase 0: 다음 대상 파악 (register.js 없는 첫 컴포넌트)
  ├ Phase 1: Agent 호출 → 단일 컴포넌트 생산 위임 (독립 컨텍스트)
  ├ Phase 2: 산출물 확인 + 커밋
  └ Phase 3: 남은 대상이 없을 때까지 반복
```

```javascript
// Phase 1 — 매 사이클 새 서브에이전트에 위임 (컨텍스트 격리)
Agent({
    description: "{컴포넌트경로} Standard 생산",
    subagent_type: "general-purpose",
    model: "opus",
    prompt: "(아래 컨텍스트 템플릿)"
});
```

서브에이전트에 주입하는 컨텍스트(프롬프트)는 자율 생산이 가능하도록 *읽을 문서·지켜야 할 제약·반환 형식*을 명시합니다. 드리프트를 막기 위해 공통 보일러플레이트를 단일 문서로 추출해 여러 스킬이 공유합니다.

```text
서브에이전트 컨텍스트 (agent-prompt-common.md 발췌)
- 먼저 읽을 문서: SHARED_INSTRUCTIONS.md → produce-component/SKILL.md
                  → COMPONENT_SYSTEM_DESIGN.md → Mixins/README.md (순서대로)
- PostToolUse hook이 차단하면 그 출력에 따라 정정 후 같은 도구를 재호출한다
  (1회 차단은 사이클 실패가 아니다)
- 신규 Mixin 생성 금지 — 기존 Mixin + 커스텀 메서드 조합으로만 해결한다
- 커밋하지 않는다 — 커밋은 메인 루프가 수행한다
- 완료 시 200단어 이내 요약만 반환 (생산 경로 / 선택 Mixin+근거 / 이벤트·구독 토픽 / 특이 결정)
```

가장 공들인 부분은 품질 보장이었습니다. 출발점은 *‘규칙을 아는 것과 규칙을 지키는 것은 다르다’* 는 관찰이었습니다. 한 차례 전수 검토에서 35개 컴포넌트 가운데 17건의 규칙 위반이 발견됐는데, 규칙은 모두 문서에 명시되어 있었음에도 구현이 그것을 따르지 않았고, 사람의 재검토조차 같은 위반을 놓쳤습니다. 그래서 품질을 사람의 감사가 아니라 **기계적 강제**로 보장하기로 했습니다. 핵심은 강제 수단을 *‘자동화가 쉬운 것’* 이 아니라 *‘위반했을 때 무너지는 것의 설계 중요도’* 순으로 정렬한 것입니다 — 아키텍처가 붕괴하는 계약, 런타임이 깨지는 계약, 생산 품질이 저하되는 계약, 스타일이 불일치하는 계약으로 등급을 나누고, 각 계약의 성격에 맞는 수단(정규식 검사, LLM 의미 검증, 파일 간 교차 검증)을 배치했습니다. 이 검사들은 에이전트가 파일을 작성하는 시점에 자동으로 실행되어, 게이트를 통과한 산출물만 유효한 것으로 인정됩니다. 전수 테스트에서 오탐 없이 실제 위반만 잡아내는 것을 확인했습니다.

운용 기준 역시 추측이 아니라 실측으로 세웠습니다. 예를 들어 생산 스킬을 더 저렴한 모델로 돌렸을 때 도메인 답습과 구조 drift가 관측되어, 해당 경로는 상위 모델로 고정하는 기준을 명시했습니다. 그 결과 사람이 매 산출물을 일일이 검토하지 않아도 일관된 품질로 컴포넌트를 대량 생산하는 파이프라인을 갖추게 되었고, 이는 AI 에이전트를 *사용하는* 것을 넘어 *설계하고 검증하고 운용하는* 역량으로 이어졌습니다.
